# feishu_ai_analysis_plugin

基于飞书多维表和电子表格，提供 LFP 批次分析、制程异常原因分析、成品数据分析、配方复盘和相似批次复盘入口。

插件定位是“AI 辅助分析”，不是自动决策系统。AI 只基于插件整理出的证据生成报告，不直接修改飞书数据、不自动放行/报废/降级、不自动改配方。

## 能力边界

- 制程异常原因分析：读取投料、湿法、喷雾、烧结、粉碎、窑炉进出窑等 Base 表，生成批次宽表和规则诊断，再由 AI 输出原因排序和措施建议。
- 成品数据自动分析：读取成品 Sheets，做表头识别、数值汇总、缺失统计、趋势摘要和数据处理建议。
- 配方和原料优化：当前做复盘和推荐方向，不直接自动优化生产配方；需要原料批次、供应商、配方版本、成品全指标和放行结果后才能升级为实验推荐。

## 口令

推荐在生产环境使用固定口令，插件也会尽量接管包含“飞书表格/制程异常/成品数据/配方复盘”等关键词的自然语言请求。

```text
批次分析 <批次号>
制程复盘 <批次号>
异常分析 <批次号> <异常现象>
成品分析 <型号或产线关键词>
成品分析 <日期> <型号或产线关键词>
配方复盘 <批次号>
配方建议 <批次号>
相似批次 <批次号>
```

## 推荐配置

```json
{
  "process_base_token": "<FEISHU_BASE_OR_WIKI_URL>",
  "product_spreadsheet_token": "<FEISHU_SHEET_OR_WIKI_URL>",
  "product_sheet_patterns": "成品数据源,型号关键词,产线关键词",
  "product_range": "A1:ZZ2000",
  "scan_limit": "500",
  "enable_ai": true,
  "llm_base_url": "<OPENAI_COMPATIBLE_BASE_URL>/v1",
  "llm_model": "<MODEL_NAME>",
  "llm_temperature": "0.2"
}
```

`llm_api_key` 按部署环境填写。若不配置 AI，插件会使用规则引擎和统计摘要兜底回复。

注意：`llm_base_url` 需要填写 OpenAI-compatible 接口根地址，并包含 `/v1`，不要只填裸域名。

## 验证建议

生产部署前建议至少完成以下验证：

- 飞书应用凭据可正常换取访问令牌。
- 制程 Base 可读，且关键表路由能匹配到现场表名。
- 成品 Sheets 可读，且表头识别、数值汇总和缺失统计正常。
- 指定批次可生成批次宽表和规则诊断。
- AI 接口可返回制程异常分析报告。

测试口令示例：

```text
异常分析 <批次号> <异常现象>
```

## 默认制程表路由

为空时使用内置路由：

- `feeding`：A/B/C/D/E 线投料汇总
- `wet_process`：A/B/C/D/E 线湿法汇总
- `spray`：A/B/C/D/E 线喷雾汇总
- `sintering`：A/B/C/D/E 线烧结汇总
- `crushing`：A/B/C/D/E 线粉碎压实汇总
- `kiln`：窑炉批次进出窑表

如现场表名不同，可配置 `process_tables_json`：

```json
{
  "feeding": ["<投料表名>"],
  "wet_process": ["<湿法表名>"],
  "spray": ["<喷雾表名>"],
  "sintering": ["<烧结表名>"],
  "crushing": ["<粉碎或压实表名>"],
  "kiln": ["<窑炉进出窑表名>"]
}
```

## AI 输入原则

插件不会把整张表发给 AI。流程是：

```text
飞书取数 -> 批次归一化 -> 批次宽表 -> 规则诊断 -> 压缩证据 JSON -> AI 报告
```

这样可以降低泄露面、减少幻觉，并保证报告有可追溯证据。

## 安全边界

- 不写入、不删除、不修改飞书原始数据。
- 不自动改配方、放行、降级或报废。
- 异常和配方建议必须标注为建议或待验证项。
- AI 调用失败时使用规则兜底，不阻塞现场查询。
